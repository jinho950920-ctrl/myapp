# [[10_쿠팡_크롤러_보안_및_인증_아키텍처]] (Coupang Crawler Auth & Security Architecture)

이 문서는 쿠팡 윙(Wing) 및 광고센터 데이터를 수집하는 모든 자동화 스크립트(`wing_sales_scraper.ts`, `test_ads_scraper.ts` 등)의 **안티봇 우회 및 세션(로그인) 운영 정책**을 규정하는 단일 진실의 원천입니다.

---

## 🛡️ 1. 세션 분리형 아키텍처 (Session Delegation)
크롤링 스크립트 본체 안에 아이디/비밀번호를 입력하고 로그인하는 하드코딩 로직을 엄격히 배제합니다. 매번 크롬 창을 새로 열어 로그인을 시도할 경우, 쿠팡 서버(Akamai)가 비정상 패턴으로 간주하여 즉각적인 IP 차단(BAN)을 내리기 때문입니다.
대신 **최초 1회 전용 로그인 봇이 획득한 쿠키를 메인 봇이 물려받아 재사용하는 구조**를 따릅니다.

*   **선발대 인증 전담 (`wing_login.ts`)**: 수동 인증 개입을 위해 창 모드로 실행되며, 쿠팡 2단계 로그인을 뚫어낸 후 `credentials/wing_session_쿠팡아이디.json` (좀비 쿠키 파일)을 생성하고 빠집니다.
*   **본대 스크래핑 전담 (`wing_sales_scraper.ts` 등)**: 이 JSON 쿠키 파일을 브라우저 렌더러에 강제 주입(Injection)하여, 로그인 화면을 아예 건너뛰고 윙 대시보드로 100% 다이렉트 진입합니다.

---

## 🤖 2. 제1단계 로그인: 100% 무인 자동 타이핑 (1st Step Automation)
쿠팡 윙의 1차 관문(아이디/비밀번호 폼 로그인)은 철저히 봇에 의한 **완전 무인화(Hands-free)** 를 보장해야 합니다.
*   기존 쿠키 만료(자연소멸, IP 변동 등)로 화면이 `/login` 창으로 돌아가더라도, 스크립트 본대가 좌절하지 않고 환경변수(`SHOPPING_ACCOUNTS`)의 아이디/비밀번호를 가져와 직접 폼에 자동 입력하고 로그인 버튼을 즉시 무인 클릭합니다.
*   아카마이 봇 방어망 패치(Stealth 설정)가 완료되어 있으므로 일 1~2회 정도의 ID/PW 1차 봇 로그인은 제재 확률이 불가능에 가깝습니다.

---

## 🛑 3. 제2단계 인증(2FA): 하이브리드 수동 개입 원칙
쿠팡의 2단계 인증(휴대폰 문자, 이메일, 앱 승인 등) 로직은 해킹 기술로 우회하려 시도하지 않습니다. 사용자(사장님)와의 **하이브리드 협업 파이프라인**을 통해 정석대로 뚫어냅니다. (이 룰이 어긋날 시 영구 계정 정지 위험이 존재함)

1. **2FA 장벽 감지 및 후퇴 (Bot)**: 본대 스크래퍼가 로그인 시도 중 2단계 인증 폼을 감지하면 즉시 크롤링을 중단합니다. 백그라운드 DB 역할을 하는 `credentials/wing_status.json` 파일의 `requires2FA` 상태를 `true`로 반전시킵니다.
2. **대시보드 경보 렌더링 (UI)**: 프론트엔드의 `[[Wing2faAlert]]` 컴포넌트(`src/components/automations/Wing2faAlert.tsx`)가 이를 실시간으로 감지하여, 대시보드 최상단에 **"쿠팡 2단계 인증이 만료되었습니다. 수동 로그인이 필요합니다."** 라는 붉은색 경고창(Alert)을 띄웁니다.
3. **수동 인증 뚫기 (Human)**: 대시보드의 알람을 확인한 사용자님께서 터미널에 `npx tsx scripts/wing_login.ts` 를 입력해 인증을 내 손으로 1회 해결해 줍니다.
4. **무인화 궤도 복귀**: 인증 직후 새로운 좀비 쿠키가 폴더에 재구워지며 알람이 해제되고, 봇은 다음 날부터 다시 1단계 무인화 모드로 영구 복귀합니다.

---

## 🐛 4. 트러블슈팅 이력 (Troubleshooting History)

### 4.1. 쿠팡 WING 맞춤 보고서 '캠페인 전체선택' 실패 트러블슈팅 (2026-04-25)
*   **이슈 증상:** 광고 데이터 매출 엑셀 다운로드 중, '캠페인을 선택하세요' 드롭다운을 열고 "전체선택" 텍스트 클릭에 성공했음에도 불구하고, 이후 엑셀 생성하기 버튼이 활성화되지 않는 문제 발생. `전체선택 클릭했지만 실제 선택 태그 0개! 재시도...` 무한 반복.
*   **원인 분석 (DOM 덤프 기반):**
    1.  **드롭다운의 정체:** 해당 컴포넌트는 일반적인 Ant Design 콤보박스가 아니라, 내부에 별도의 체크박스 배열을 품고 있는 **커스텀 리액트(React) 팝오버 컴포넌트**였음.
    2.  **React 이벤트 무시:** 이전 코드에서는 `page.evaluate()`를 사용해 자바스크립트의 기본 DOM 이벤트(`element.click()`)를 직접 주입했음. 시각적으로는 체크된 것처럼 보였으나, 쿠팡 페이지의 React 프레임워크가 이 이벤트를 감지하지 못해 드롭다운을 닫는 순간 선택 상태가 증발함.
    3.  **오작동한 검증 로직:** 드롭다운이 닫힌 후 `.ant-select-selection-overflow-item` 요소의 개수로 선택 성공 여부를 검증했으나, 이 커스텀 컴포넌트는 해당 요소(태그)를 생성하지 않기 때문에 항상 0개로 오판함.
*   **해결 조치 (`[[wing_sales_scraper.ts]]` 수정 사항):**
    *   **[수정 전]** 순수 JS `evaluate` 구문으로 DOM을 직접 클릭하고 강제로 이벤트 발생 시도.
    *   **[수정 후]** 사람의 실제 마우스 조작을 에뮬레이트하는 **Playwright 네이티브 Locator**로 교체하여 React의 `SyntheticEvent`가 정상적으로 트리거되도록 수정.
    ```typescript
    // 방법 1: Playwright locator로 label 직접 클릭 (React 이벤트 정상 트리거)
    const selectAllLabel = page.locator('label.ant-checkbox-wrapper').filter({ hasText: /전체\s*선택/ }).first();
    if (await selectAllLabel.isVisible({ timeout: 2000 }).catch(() => false)) {
      await selectAllLabel.click({ force: true });
      
      // 검증: 드롭다운 내부의 실제 input 상태 확인
      const isChecked = await selectAllLabel.locator('input[type="checkbox"]').isChecked().catch(() => false);
      if (isChecked) clickedSelectAll = true;
    }
    ```
